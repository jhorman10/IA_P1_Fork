## Context

The appointment management system needs persistent storage for:
- **Core Appointment Data**: ID, date, patient, doctor, room, status
- **Metadata**: Custom fields, preferences, notes, attachments
- **Audit Trail**: Creation/modification timestamps, who created it
- **Query Patterns**: Find by date range, by doctor, by patient, by room, by status

### Schema Flexibility
Different institutions have different needs:
- Hospital A: Requires "emergencyContact" field
- Hospital B: Requires "insuranceCode" field
- Hospital C: Requires custom "departmentCode" field

Rigid schema (SQL) would require:
- Database migration for each customer
- ALTER TABLE statements
- Coordination with operational team
- Potential downtime

## Alternatives Considered

### 1. PostgreSQL (SQL Relational Database)
```sql
CREATE TABLE appointments (
  id UUID PRIMARY KEY,
  id_card VARCHAR(20) NOT NULL,
  full_name VARCHAR(255) NOT NULL,
  date TIMESTAMP NOT NULL,
  office VARCHAR(100),
  status VARCHAR(50),
  
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_date ON appointments(date);
CREATE INDEX idx_id_card ON appointments(id_card);
-- Add new field = ALTER TABLE + migration
```

**Pros**:
- ACID compliance fully guaranteed
- Mature ecosystem
- Strong data integrity
- Powerful SQL features
- Good for relational queries (JOIN across tables)

**Cons**: ❌
- Strict schema (all columns must exist)
- Schema changes require migrations
- Adding optional fields = NULL columns (waste)
- Overkill for semi-structured data
- Slower for document-oriented queries
- More overhead for distributed systems

**Decision**: Rejected - Schema inflexibility contradicts requirement for multi-tenant customization

```typescript
// infrastructure/repositories/mongo-appointment.repository.ts
@Injectable()
export class MongoAppointmentRepository implements AppointmentRepository {
  constructor(
    @InjectModel('Appointment')
    private appointmentModel: Model<AppointmentDocument>
  ) {}
  
  async save(appointment: Appointment): Promise<void> {
    const doc = this.mapToDocument(appointment);
    await this.appointmentModel.updateOne(
      { _id: doc._id },
      doc,
      { upsert: true }
    );
  }
  
  async findById(id: AppointmentId): Promise<Appointment | null> {
    const doc = await this.appointmentModel.findById(id);
    return doc ? this.mapToDomain(doc) : null;
  }
  
  async findByDateRange(start: Date, end: Date): Promise<Appointment[]> {
    const docs = await this.appointmentModel
      .find({ date: { $gte: start, $lt: end } })
      .sort({ date: 1 })
      .exec();
    
    return docs.map(doc => this.mapToDomain(doc));
  }
```

```yaml
volumes:
  mongo_data:
    driver: local
```

## Benefits and Consequences

### Positive Consequences ✅

1. **Schema Flexibility**
   - Add fields without migration
   - Optional fields naturally represented (document)
   - Each institution can have custom metadata
   - **Example**: Hospital A adds "emergencyContact", Hospital B adds "insuranceCode"
   - No ALTER TABLE required

2. **Developer-Friendly**
   - JSON documents = natural JavaScript/TypeScript
   - No impedance mismatch (ORM)
   - Fast prototyping
   - Intuitive querying

3. **Rich Querying**
   - Complex filters: `{ status: 'confirmed', date: { $gte: ... } }`
   - Aggregation pipeline: complex transformations in database
   - Text search: full-text indexing
   - Geospatial: location-based queries

4. **Horizontal Scalability**
   - MongoDB sharding distributes data across servers
   - Supports growth from 1K to 1B appointments
   - Replica sets for HA (automatic failover)

5. **Atomic Transactions** (4.0+)
   - ACID compliance within single db
   - Handles distributed scenarios (eventual consistency)

6. **Ecosystem**
   - Mongoose ODM with validation
   - Good tooling (MongoDB Compass GUI, mongo shell)
   - Large community and libraries
   - Cloud option (MongoDB Atlas)

### Negative Consequences ⚠️

1. **Eventual Consistency**
   - Replication copies data asynchronously
   - Brief period where replicas lag primary
   - **Mitigation**: Read from primary for critical operations, acceptable for appointment system

2. **Requires Manual Index Management**
   - Must anticipate common queries
   - Wrong indexes = slow queries
   - **Mitigation**: Schema design phase, performance testing

3. **Storage Overhead**
   - Document format uses more space than normalized SQL tables
   - Redundancy across replicas
   - **Mitigation**: Compression, archival of old data

4. **Lack of Referential Integrity**
   - Foreign keys not enforced by database
   - Orphaned documents possible
   - **Mitigation**: Application-level validation (domain layer)

5. **Large Result Sets**
   - Documents can be large (all metadata embedded)
   - Must paginate for large result sets
   - **Mitigation**: Projection (select only needed fields), pagination

6. **Normalization Difficult**
   - Duplicating data across documents is common
   - Updates might require multiple documents
   - **Mitigation**: Design for document boundaries, use transactions
